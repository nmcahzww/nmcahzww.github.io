搭建博客的记录   blog org @blog

我比较懒，很久之前折腾过一次博客，但是养不成记录的习惯就荒废了，这次想重新折腾一下..

hugo 之前

这次是用了 hugo 来搭建博客，搭配 github page 非常简单。之前尝试过 hexo, 还专门配了 travis 做部署的事情。

但是 hexo 现在主题仓库几乎都是 Axxx 这种命名来抢位置，导致想找一款适合的主题很难，要么从前几页选，要么用大家公认的好的。 我就是选了 next, 算简洁的了，但是用了一阵子还是不喜欢，不够简单。再加上在网上看到一些博客纷纷从 hexo 迁移到 hugo, 似乎是嫌弃 hexo 太慢， 虽然我不觉得这是什么问题，但是还是决定尝鲜一下。

hugo + even

之前试过别人推荐的主题 jane, 但是下方的图标 size 错乱让我很难受，我又不想改 css，于是就用了 jane 所参考的 even 主题。

ox-hugo

emacs org mode 太强了，写博客这种事情太适合 org 了，于是找到了这个 package 来做 org -> md 的转换。

这个包支持 subtree 的形式，就是一个 org 文件中的每个 subtree 对应一篇 md， 所以理论上整个博客可以只维护一个 org 文件，但是他也支持 org -> md 一对一的形式。

维护 tag 和 category 几乎和 org 原生的一样，很方便。todo item 可以设置文章的状态，默认配置是状态为 todo 的文章是不渲染的。

具体的写作流程是：

  • hugo 相关配置，加入类似 HUGO_SECTION, HUGO_BASE_DIR 的参数，这个只需做一次

  • 打开相应的 org 文件，找到相应的 subtree 位置

  • 写文章，配好需要的 tag, category, export_date 等

  • 写完之后 org export 一下，选择对应的 option, 然后就会生成 hugo 需要的 md 文件在对应的目录

然后写作部分就完成了。

deploy

这里最后需要一步部署的工作，这里只提供部署到 github page 相关的参考。

直接运行 hugo 会把生成的静态文件放在 /public 文件夹内，其实只需要把 /public 里的内容 push 到 github page 仓库的 master 分支就可以。 为了方便的话，可以把 /public 目录设置成 git submodule, origin 是 github page 仓库， 然后整个 blog 目录是另一个仓库，然后写个脚本管理就可以了：

  #!/bin/sh

  # If a command fails then the deploy stops
  set -e

  printf "\033[0;32mDeploying updates to GitHub...\033[0m\n"

  # Build the project.
  hugo -t even # if using a theme, replace with `hugo -t <YOURTHEME>`

  # Commit changes.
  msg="rebuilding site $(date)"
  if [ -n "$*" ]; then
          msg="$*"
  fi

  push() {
      git add .
      if [ $msg == 'clean' ]; then
          git commit --amend --no-edit
          git push origin master -f
      else
          git commit -m "$msg"
          git push origin master
      fi
  }

  cd public && push
  cd .. && push

Emacs as Python IDE   emacs python @emacs

从用 emacs 作为 python 主力的开发环境已经有一阵子了，最近也是因为新的 python 项目折腾了挺久配置，打算记录下这些过程，就想到啥说啥吧。

初识 Emacs

我是从 2018 年 6 月开始写代码的，最开始对编程语言、ide、编辑器等等是完全没有概念的。鉴于当时情况，工作中使用 python， 所以最开始用 pycharm， 不久后换成了 vscode。当时我的 mentor 是深度 emacs 使用者，从他那边算是对 emacs 有个简单的认知： emacs 是个能干很多事的编辑器 。那时候我还是受了很多他的影响的， 特别是一些常用的 key bindings，所以不知不觉中，其实已经熟悉了 emacs 的键位。

后来大约在 2019 年 3 月，我有了想试试 emacs 的念头，现在想起来面对同事圈子内庞大的 vim 党，没选择 vim 完全是因为我两个都不会，选了个脸熟的。 从那开始度过了安装、过 tutorial 等等，算是能正常的 开启和关闭 使用 emacs 了。那段时间看了很多网上的视频和论坛，打开了配置 emacs 的大门， 就一直在折腾一些很搞笑的事情，这个好看那个好用什么的，还挺好玩的。

打造 Python IDE

到了 2019 年夏天吧，一些简单基本的都折腾累了，那个时候我的 emacs 单纯从编辑体验上来说，相比与 vscode 其实没什么差距，就是好看一点以及更好的 key bindings。 所以就想把平时写代码的环境挪到 emacs 上。当然 org mode 排除在外… org 真的太好用太强大了。

elpy

最早我用的是 elpy 这个包，算是那段时间比较主流的包，主要是提供对语言的基本支持，例如补全跳转之类的。虚拟环境还是用 virtualenvwrapper，最后配合 flycheck 做代码检查。 最初的情况就这么简单，但是老实说，体验不太好，主要的痛点在于：编辑单个文件的效率有了，但是项目中的 quick navigation 没有，例如项目中的全局搜索、快速 look up 一个文件等。

lsp

后来被安利了 lsp，替换掉了原来的 elpy。这里是没有解决刚刚的问题的，只是单纯的语言 parse 能力要强很多，这个阶段主要是 lsp-mode + flycheck + company ， 基本上单个编辑工作流（编辑一个文件 -> 根据引用跳转到另一个文件）已经是能有很好的效率了。

projectile & neotree

引入 projectile 和 neotree 就是为了解决项目和路径树的问题，projectile 真的挺好用的，一般都会有多个不同的项目需要管理，包括在项目内部也支持很多操作。neotree 主要就是 在面对大型项目，或者是习惯了 ide 文件目录那种视图的话，比较方便的看路径结构，虽然也许对一个项目熟了之后可能不需要这个，但是我觉得还是能减少大脑思考的时间。

magit

不多说，killer package

Emacs & Vscode

经过上面的几次变动后，emacs 目前在 python 项目开发中，具有了：

  • 项目间、项目中的各类操作支持（切换、检索）

  • 语言支持（提示、补全、检测） 这些能力

基本上项目的开发过程是包括了的，我个人如果项目本地运行的话还是习惯在终端里跑，或者在开发机之类，所以 emacs 配置没有包含运行程序的部分，当然 emacs 也可以 呼出 shell 或者 ipython 之类的，但是体验并不好，还是终端好用…

之前逛 emacs 的论坛，会看到有很多人会说转 vscode 了。我个人对这两个其实不想做什么比较的，因为本质上大家都是成年人了，可以都要啊… 这两个不是非此即彼的关系。 例如我写 js 可能就用 vscode 了，或者我用了一阵子 emacs 也会用 vscode 一段时间。当然最关键的，我觉得还是这两者定位本身就不同，至少在我这不同：

  • emacs 是个让你写东西很爽的东西，能做很多事（写博客、折腾），刚巧你也能通过配置来很舒服的 coding

  • vscode 就是个很好用的写代码的工具，有很多 emacs 没有的东西，例如更炫酷的特效，更多的不需要折腾的扩展，更容易上手的体验

所以目前我双修：）

最近折腾的东西

其实虽然我折腾了一些时间，但是老实说平时空闲时间很少，现在的配置基本属于刚刚够用。我其实很懒，所以能用我基本上不会折腾（除非发现了很炫酷的东西）。

python2.7 的问题

最近是因为新的项目年代久远是 python2.7 写的，还有一些运行环境的不同，所以我之前那份配置的 flycheck 有点问题。 flycheck 是我的主力代码检测工具，checker 设置的是 pylint，但是 flycheck 默认的路径是 /usr/bin/python3 ，所以导致 pylint 不可用。

我有多个项目和多个虚拟环境，我需要在不同的虚拟环境下，flycheck 能识别到对应的 pylint executable。我之前写 python3 项目的时候， 我印象中是没有专门配置过这个的，而且我的 pylint 工作的非常好，所以这里我僵住了，不知道为什么突然这样。所以这里在我的配置中加了一些代码：

(use-package virtualenvwrapper
  :ensure t
  :init
  (setq venv-workon-cd t)
  (add-hook 'venv-postactivate-hook (lambda ()
    (setq flycheck-python-pylint-executable (concat "/Users/nathan/.virtualenvs/" venv-current-name "/bin/pylint")))))

第二个是，之前的配置加载逻辑有点乱，有时候会出现 lsp 没启动但是 flycheck 已经加载了，后来仔细的调整了几个配置的顺序：

virtualenvwrapper -> flycheck set executable; python-mode -> run lsp & enable flycheck pylint

后记

在上大学的时候，觉得视频承载的信息量是文字、声音等比不了的，对于碎片化的时间，只有足够信息密度的载体，才更容易占据。那时候我的很多空余时间都花在了看电影或者看剧上面， 在那样的环境里，老实地说，我没有定力去啃书。

另外看书的另一个实在的感受，是焦虑。仿佛看了书就走的慢了，就会和社会脱节。仿佛人类都在奔涌向前，左顾右看探索着永远无尽的地域，而你选择闭上眼睛。但是后来也想通了， 也许正是当别人都走在前面时，才需要有人走在后面。希望能坚持，能静下心来生活。

大家都说工作决定生活，有时候想想生活才决定工作啊，还是要努力丰富生活。

工作

最近在考虑换工作的事情，想要为自己寻找下一个合适的平台，导致生活工作节奏都紧了许多。 这是我转行做程序员以来考虑得最久也是最谨慎的一件事了，每天除了工作上的内容，需要花很多时间去学习和准备找工作需要的东西。 给自己定的计划是大约在5、6月份需要找到合适的工作，到今天为止已经开始准备大概半个月了，所以记录一下。

学习

这里的学习非常的肤浅，主要还是为了面试的知识补充。对于非科班的人来说，知识的积累不是系统性的，可能会有很多问题。 最近主要在学习算法一块，后面可能还要补很多计算机的基础知识。

算法主要是上了网上的一些公开课，我上了 mit6006 的基础课，youtube 或者 b站 上都有， 我看的是 Erik Demaine 助教的版本，比较喜欢他的风格。主要是为了系统地学一下数据结构和部分经典的算法，例如树或者 dp 等。

刷题

哎面试总是逃不掉这个，没有竞赛经验，没有系统地上过算法课，这方面总是要吃力点。刷题本身我很讨厌， 后来仔细的想想，还是得 play by the rules. 所以就开始痛苦的刷题了… 主要还是 leetcode，到今天为止一共做了300题左右，但是其中有部分题是很久之前做的， 还有部分是看 discuss 完成的，所以也不能算“刷”吧。

现在基本上保证了平均每天10道左右的速度，工作日会少一些，周末会补回来这样，然后从上周开始第一次参加了周赛，做了3道。 做题真的是经验性的行为，特别是算法题，我现在特别明显的感觉到做多了之后和我开始刷题之前的差距，而这才过了十几天。 很多题也 post 了自己的一些解法和想法，其实做多了之后发现做题本身是不怎么需要智商的行为。

目前刷题的打算是刷到月底就差不多了，感觉 medium 的差不多能在压力环境下半个小时 bug free 就可以了。后面看要不要针对面试的公司再去做一些 hard 的题。 按目前的情况，这个计划问题应该不大。

生活

在开始找工作的计划之后，虽然休息的时间变得很少，可以说几乎没有，但是不知道为什么有点乐在其中的感觉。不管是上课还是刷题， 感觉应该是收到的正反馈还算比较可以，并且自己心里清楚这是件好事吧。我的自制力相对不强的，但是这次连我自己都有点意外， 对没错，坚持半个月已经算“意外”了..

希望会有好结果吧，然后会再来更新。
